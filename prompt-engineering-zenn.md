「面白きこともなき世を面白く」をモットーに、IT研修の現場でつまずきやすいところを噛みくだいてお届けしています。

今日のテーマは **プロンプトエンジニアリング**。
「AIに上手にお願いするコツでしょ？」と思われがちですが、本質はもう少し深くて、**良い結果を“偶然”ではなく“再現可能”にする技術**です。

この記事では、わたしが実際にやった「スライドのデザインを“再現プロンプト”にまとめる」という作業を題材に、初心者の方でも今日から使える型を紹介します。

---

## WHY ― なぜプロンプトエンジニアリングが必要なのか

先日、研修資料のスライドをAIと一緒に作りました。配色も雰囲気も気に入ったものができたのですが、ここで問題が起きます。

> 次回、別テーマで同じ雰囲気のスライドを作りたい。でも、あの時どうお願いしたか覚えていない。

これ、現場あるあるです。AIは毎回まっさらな状態から答えるので、**こちら側の指示が曖昧だと、出力も毎回ブレます**。たまたま良いものが出ても、それは「運」であって「実力」ではありません。

そこでわたしは、あの傑作を生んだ条件を1枚の文書にまとめました。それが **デザイン再現プロンプト** です。
この「再現できる形に言語化する」行為こそが、プロンプトエンジニアリングの正体です。

---

## WHAT ― プロンプトエンジニアリングとは

**定義**：AIから望む出力を安定して引き出すために、指示（プロンプト）を設計・改善する技術。

**たとえ（うさうさラーメン店）**：
AIをラーメン店の新人さんだと思ってください。

- 「いい感じのラーメン作って」→ 何が出てくるか分からない（=曖昧なプロンプト）
- 「醤油ベース、麺は細麺かため、チャーシュー2枚、ネギ多め、提供は3分以内」→ 毎回同じ一杯が出る（=設計されたプロンプト）

腕のいい新人さんでも、注文が雑なら良い一杯は“偶然”しか出ません。注文を整えるのは、お客さん側の仕事なのです。

**詳細**：プロンプトエンジニアリングは、役割の指定・制約の明示・例の提示・構造化など、いくつかの再現可能なテクニックの組み合わせでできています。次の章で具体的に見ていきます。

**一言まとめ**：プロンプトエンジニアリングとは、「運で出た傑作」を「毎回の品質」に変える注文の技術。

## HOW ― 今日から使える7つの型

それぞれ「悪い例 → 良い例」で見ていきます。

### 1. 明確で具体的にする（曖昧さを消す）

AIは空気を読めません。形容詞ではなく、判断できる基準で書きます。

```text
❌ おしゃれな色にして
⭕ 主役の色は深いワインローズ（hex: 5C2438）、差し色はローズ（A23E5C）にして
```

「おしゃれ」は人によって違いますが、hexコードは誰が見ても同じです。**数字・固有名詞・基準値に置き換える**のがコツ。

### 2. 役割を与える（ロールプロンプト）

冒頭で「誰として答えるか」を決めると、語彙や判断基準が引き上がります。

```text
⭕ あなたは一流のエンジニア兼デザイナーです。法人研修向けの資料を作ってください。
```

「一流のデザイナー」と名乗らせるだけで、配色や余白への配慮が変わります。

### 3. 制約・禁止事項を明示する

「やってほしいこと」と同じくらい、「やってほしくないこと」が効きます。

```text
⭕ 禁止：見出しの下線、アクセントの帯、本文の中央寄せ、クリーム/ベージュ地。
   個人情報（人名・メール・地名）は出さず、登場人物は役割名のみ。
```

AIが陥りがちな“それっぽいけどダサい”パターンを、先回りで塞いでおきます。

### 4. 例を見せる（Few-shot）

言葉で説明しづらいものは、見本を1〜2個渡すのが最速です。

```text
⭕ 出力はこの形式で：
   見出し（定義）→ たとえ → 詳細 → 一言まとめ
   例）「PEST分析とは…（定義）。天気予報のように…（たとえ）。具体的には…（詳細）。つまり、未来の追い風と向かい風を読む地図（一言）」
```

「こういう感じ」を1回見せると、以降ずっとその型で返ってきます。

### 5. 構造化する（見出し・タグで区切る）

長い指示は、ベタ書きより**区切る**と精度が上がります。Claudeなどでは見出しやXMLタグが特に有効です。

```text
⭕
# 役割
あなたは…

# 配色
- ワインローズ 5C2438（主役）
- ローズ A23E5C（差し色）

# 禁止事項
- …
```

人間が読みやすい構造は、AIにとっても読みやすい構造です。

### 6. 段階的に考えさせる（ステップ指定）

複雑なタスクは、いきなり完成形を求めず手順を踏ませます。

```text
⭕ 次の順で進めて：
   1. まず構成（WHY→WHAT→HOW）を箇条書きで提案
   2. OKを出したら各スライドの文言を作成
   3. 最後にデザイン仕様を適用
```

「一度に全部」より「順番に」のほうが、後戻りが減ります。

### 7. 出力フォーマットを指定する

そのまま使える形で受け取ると、こちらの手間が消えます。

```text
⭕ 出力はZenn記事用のMarkdown（frontmatter付き）で。コードブロックは```で囲んで。
```

## 実例 ― 「デザイン再現プロンプト」を解剖する

冒頭で触れた再現プロンプトは、上の型をぎゅっと詰め込んだものです。抜粋するとこんな構造になっています。

```text
あなたは一流のエンジニア兼デザイナーです。…   ← ② 役割
16:9ワイド（13.333 × 7.5 inch）で…            ← ① 具体的な数値
配色（必ずこのhexを使用・#は付けない）
  ワインローズ 5C2438 / ローズ A23E5C / …     ← ① 基準の固定
共通モチーフ：アイコンは必ず淡い丸の中に置く   ← ④ 再現したい特徴を言語化
禁止：見出し下線、帯、本文中央寄せ…            ← ③ 禁止事項
構成は WHY→WHAT→HOW、説明は4段で…           ← ⑤⑥ 構造と手順
テーマは【ここに内容】                         ← 差し替え変数
```

ポイントは最後の **`【ここに内容】` という差し替え変数**。
テーマだけ入れ替えれば、同じ世界観のスライドが何度でも作れます。一度の傑作が、再利用できる“型”に昇格した瞬間です。これがプロンプトエンジニアリングのうれしさ。

## まとめ

| 型 | ひとことで |
|---|---|
| ① 具体的に | 形容詞を数字・固有名詞に |
| ② 役割を与える | 「一流の◯◯として」 |
| ③ 制約を示す | 禁止事項こそ効く |
| ④ 例を見せる | 見本1つは説明100行に勝る |
| ⑤ 構造化する | 見出し・タグで区切る |
| ⑥ 段階化する | 一度に全部、をやめる |
| ⑦ 形式指定 | そのまま使える形で受け取る |

プロンプトエンジニアリングは、特別な才能ではなく**注文を整える習慣**です。
良い結果が出たら、そこで終わらせず「どうお願いしたから良かったのか」を1枚にまとめてみてください。あなたの傑作が、明日からの定番になります。

それでは、面白きこともなき世を面白く。
